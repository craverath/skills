Skills em tópicos

using-superpowers Meta

Porta de entrada · sempre ativa
  • Ativa no início de qualquer conversa, antes de responder.
  • Regra do "1%": se houver chance de uma skill se aplicar, invoca para checar.
  • Prioridade: instruções do usuário > skills > prompt padrão.
  • Skills de processo primeiro (brainstorming, debugging), depois implementação.
  • Anuncia a skill em uso e a segue à risca.

brainstorming Colaboração

Etapa 1 · transforma ideia em design
  • Diálogo socrático: uma pergunta por vez, múltipla escolha quando dá.
  • Propõe 2-3 abordagens com prós/contras e recomendação.
  • Apresenta o design em seções, aprovando cada uma.
  • Nenhum código antes do design aprovado — vale até para projeto "simples".
  • Salva o spec em docs/ e pede revisão antes de seguir.

using-git-worktrees Colaboração

Etapa 2 · espaço de trabalho isolado
  • Detecta isolamento já existente antes de criar qualquer coisa.
  • Prefere ferramentas nativas; git worktree só como fallback.
  • Verifica que o diretório está no .gitignore.
  • Roda o setup do projeto (npm/cargo/pip…).
  • Confirma um baseline de testes limpo antes de implementar.

writing-plans Colaboração

Etapa 3 · design em plano de tarefas
  • Plano para alguém sem contexto: caminhos de arquivo exatos.
  • Tarefas pequenas (2-5 min) com código e passos de verificação.
  • Princípios: DRY, YAGNI, TDD, commits frequentes.
  • Proibido placeholder ("TODO", "depois", "tratar erros").
  • Autorrevisão de cobertura do spec e consistência de tipos.

subagent-driven-development Colaboração

Etapa 4 (recomendada) · construir tarefa a tarefa
  • Um subagente novo por tarefa, com contexto isolado.
  • Revisão em duas etapas: conformidade com o spec, depois qualidade.
  • Execução contínua — não pausa para checar entre tarefas.
  • Usa o modelo mais barato que dá conta de cada papel.
  • Trata status do implementador: DONE, BLOCKED, NEEDS_CONTEXT…

executing-plans Colaboração

Etapa 4 (alternativa) · sem subagentes
  • Executa o plano em sessão separada, com checkpoints.
  • Revisa o plano criticamente antes de começar.
  • Segue cada passo à risca e roda as verificações.
  • Para e pergunta quando bloqueado — não chuta.
  • Encerra chamando finishing-a-development-branch.

dispatching-parallel-agents Colaboração

Apoio · paralelismo
  • Para 2+ tarefas independentes, sem estado compartilhado.
  • Um agente por domínio de problema, rodando em paralelo.
  • Prompts focados, autocontidos, com saída esperada clara.
  • Revisa, checa conflitos e integra; roda a suíte completa.
  • Não usar quando as falhas são relacionadas entre si.

test-driven-development Testes

Dentro da implementação · teste primeiro
  • Lei de ferro: nenhum código de produção sem um teste que falhe antes.
  • Ciclo VERMELHO → VERDE → REFATORAR.
  • Ver o teste falhar prova que ele testa a coisa certa.
  • Código mínimo para passar; nada de over-engineering.
  • Escreveu código antes do teste? Apaga e recomeça pelo teste.

requesting-code-review Colaboração

Entre tarefas · pedir revisão
  • Despacha um revisor com contexto preciso (não o histórico da sessão).
  • Revisa cedo e com frequência.
  • Classifica problemas: Crítico / Importante / Pequeno.
  • Corrige críticos e importantes antes de seguir.
  • Rebate o revisor com argumento técnico quando ele erra.

receiving-code-review Colaboração

Resposta · receber feedback
  • Avaliação técnica, não concordância performática.
  • Proibido "você tem toda razão!", "ótimo ponto!".
  • Verifica contra a realidade do código antes de implementar.
  • Esclarece todos os itens dúbios antes de começar.
  • Rebate com razão técnica e aplica YAGNI.

systematic-debugging Depuração

Quando surge um bug
  • Lei de ferro: nenhuma correção sem investigar a causa raiz.
  • 4 fases: causa raiz → padrão → hipótese → implementação.
  • Reproduz, lê erros, checa mudanças recentes, instrumenta.
  • Uma hipótese e uma mudança por vez.
  • 3+ correções falharam? Questione a arquitetura.

verification-before-completion Depuração

Antes de concluir qualquer coisa
  • Lei de ferro: nenhuma alegação de "pronto" sem evidência fresca.
  • Rode o comando, leia a saída, só então afirme.
  • "Deveria funcionar" / "provavelmente" são bandeiras vermelhas.
  • Não confie em relatório de subagente — verifique o diff.
  • Evidência antes de alegações, sempre.

finishing-a-development-branch Colaboração

Etapa 5 · entregar o trabalho
  • Verifica que os testes passam antes de apresentar opções.
  • Detecta o ambiente: repo normal / worktree / detached HEAD.
  • 4 opções: merge / PR / manter / descartar.
  • Descartar exige confirmação digitada.
  • Limpa o worktree só nos casos certos (merge e descartar).

writing-skills Meta

Criar/editar as próprias skills
  • Escrever skills É TDD aplicado a documentação de processo.
  • Cenário de pressão = teste; SKILL.md = código de produção.
  • Veja o agente falhar sem a skill (baseline) antes de escrever.
  • description = "quando usar", nunca resumo do processo (CSO).
  • Refatora fechando brechas e racionalizações.
