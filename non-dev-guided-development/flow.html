Desenvolvimento guiado

Um fluxo passo a passo onde você descreve o que quer e o agente cuida da técnica, explicando cada etapa em linguagem simples.

Baseado na metodologia Superpowers, simplificada para quem não programa. Regras globais desta pasta em AGENTS.md.

O fluxo em 5 passos

Toda ideia percorre estes passos, na ordem. Nada é pulado por "parecer simples", e cada decisão importante é aprovada por você antes de seguir.

PASSO 1
brainstorming
Transformar a ideia em um desenho aprovado.
PASSO 2
writing-plans
Virar o desenho em um plano de passos pequenos.
PASSO 3
test-driven-development
Construir cada parte com testes primeiro.
PASSO 4
requesting-code-review
Revisão independente antes de entregar.
PASSO 5
finishing-a-development-branch
Conferir testes e entregar com sua confirmação.

O que cada etapa faz

1. brainstorming

Entrada: uma ideia solta · Saída: docs/specs/AAAA-MM-DD--design.md aprovado
  • Conversa com você uma pergunta por vez, de preferência em múltipla escolha.
  • Propõe 2–3 abordagens (escopo/desenho) — a linguagem já é fixa pela stack.
  • Sinaliza o que é sensível (login, pagamento, dados pessoais, upload).
  • Não escreve código nem cria arquivos antes do desenho ser aprovado.

2. writing-plans

Entrada: desenho aprovado · Saída: docs/plans/AAAA-MM-DD--plan.md aprovado
  • Quebra o trabalho em tarefas pequenas e verificáveis (poucos minutos cada).
  • Cada tarefa diz o que será feito e como saber que deu certo.
  • Sem lacunas: nada de "tratar erros depois" sem dizer concretamente o quê.
  • Confere o plano contra o desenho (cobertura, lacunas, coerência) antes de seguir.

3. test-driven-development

Entrada: plano aprovado · Saída: código funcionando com testes passando
  • Ciclo vermelho → verde → limpar: teste primeiro, vê falhar, constrói o mínimo, vê passar, organiza.
  • Nenhuma parte é construída sem um teste que falhe antes.
  • Bug? Primeiro um teste que o reproduz, depois o conserto — fica registrado para sempre.
  • Tudo tipado e na stack obrigatória (TypeScript ou Python).

4. requesting-code-review

Entrada: funcionalidade pronta · Saída: problemas classificados e resolvidos
  • Um revisor independente compara o que foi feito com o que o plano pedia.
  • Classifica os problemas: Crítico · Importante · Pequeno.
  • Passo de segurança obrigatório — toda falha conta como crítica/importante.
  • Críticos e importantes são resolvidos antes de entregar; pequenos ficam anotados.

5. finishing-a-development-branch

Entrada: trabalho revisado · Saída: entrega feita com sua confirmação
  • Roda todos os testes; se algum falha, para e mostra o que quebrou.
  • Apresenta opções claras: juntar, abrir Pull Request, deixar como está, ou descartar.
  • Descartar exige confirmação digitada (você escreve discard).
  • Nunca faz commit/push/merge sozinho — só com sua confirmação explícita.

Regras que valem em todo o fluxo

🔒 Você sempre aprova

  • O agente não pula etapas, mesmo em projetos "simples".
  • Cada decisão importante é apresentada para você aprovar antes de seguir.
  • Nada de commit, push ou merge sem sua confirmação explícita.

🧪 Sempre com testes e tipos

  • Toda função nova tem teste; todo bug vira um teste de regressão.
  • Tipagem obrigatória — sem any em TS, sem função sem tipo em Python.
  • "Testei na mão" não basta: o teste fica registrado e repetível.

🗣️ Linguagem simples

  • Você não precisa entender o código — o agente traduz tudo.
  • Sem jargão; termos técnicos vêm com uma explicação curta.
  • O agente conversa no seu idioma (os prompts são em inglês, a conversa não).

🛡️ Segurança por padrão

  • Segredos nunca no código; .env fora do repositório.
  • Toda entrada externa é validada; acesso ao banco sem injeção.
  • Permissões conferidas no servidor; dependências auditadas.

Tecnologia: a stack é fixa

Você não escolhe a linguagem — ela é definida pelo tipo de projeto. Só TypeScript ou Python, sempre tipado e testado.

🌐 Site ou app no navegador

Páginas, painéis, login, dashboards, lojas…

Next.js + TypeScript
pnpm · Vitest + Playwright · ESLint + Prettier · TS estrito (sem any)

⚙️ Script, automação ou CLI

Processar arquivos/dados, integrar serviços, tarefas agendadas…

Python
uv · pytest · pyright (estrito) · ruff · type hints completos

🖥️ Programa de janela (desktop)

Aplicativo com interface que abre no computador.

Python ou Electron
Electron: TS + React · Python: PySide6 — veja o desempate abaixo

Desempate de desktop: Python ou Electron?

Prefira Electron quando…Prefira Python quando…
A interface é rica, visual ou parecida com um site. O programa é mais sobre processar dados/arquivos do que sobre a tela.
Você reaproveita conhecimento/código web (Next.js). Há muita integração com sistema operacional, hardware ou ferramentas locais.
Distribuir um instalador bonito multiplataforma importa. A interface é simples e a instalação enxuta importa.

Segurança por stack

Além das regras gerais (segredos, validação, sem injeção, menor privilégio, dependências auditadas), cada base tem cuidados próprios.

🌐 Web — Next.js

  • Auth com biblioteca confiável — nunca sessão/criptografia caseira.
  • Permissão conferida no servidor; esconder a tela não é controle.
  • Cookies httpOnly+secure; proteção contra CSRF e XSS; CSP definida.
  • Segredos só no servidor — nunca em NEXT_PUBLIC_*.

⚙️ Python

  • Validação com pydantic; consultas ao banco parametrizadas.
  • Sem eval/exec ou pickle em dados não confiáveis.
  • subprocess sem shell=True com entrada externa.
  • Proteção contra path traversal ao lidar com arquivos.

🖥️ Electron

  • contextIsolation e sandbox ligados; nodeIntegration desligado.
  • API mínima via contextBridge; toda mensagem IPC validada.
  • Carregar só conteúdo local confiável; CSP definida.
  • Nunca executar código remoto nem desligar webSecurity.

Como configurar no seu ambiente

Você precisa do Claude Code instalado. As skills e as regras ficam disponíveis em qualquer projeto seu — configura uma vez.

  1. Instalar as skills (uma vez, vale para todo projeto)

    Copie as 5 pastas de skill para a pasta pessoal do Claude Code. A partir da raiz deste repositório:

    # cria a pasta de skills pessoais e copia as 5 skills
mkdir -p ~/.claude/skills
cp -R non-dev-guided-development/*/ ~/.claude/skills/

    Pronto: brainstorming, writing-plans, test-driven-development, requesting-code-review e finishing-a-development-branch passam a existir em qualquer projeto.

  2. Ligar as regras por padrão (stack, segurança, estilo)

    Acrescente o conteúdo das regras globais ao arquivo de memória do Claude Code. Assim o agente já segue tudo sozinho, sem você precisar pedir:

    cat non-dev-guided-development/AGENTS.md >> ~/.claude/CLAUDE.md

    É o que garante "só TypeScript ou Python", "sempre tipado e testado" e os pontos de segurança em todo projeto.

  3. Ferramentas de cada stack (instala quando o projeto pedir)

    O agente avisa o que falta. Em geral: Node.js + pnpm para web/Electron, e Python + uv para scripts.

    # pnpm (via Node/corepack)
corepack enable pnpm
# uv (gerenciador de Python)
curl -LsSf https://astral.sh/uv/install.sh | sh

  4. Conferir

    Abra o Claude Code dentro da pasta do seu projeto e digite / — as 5 skills aparecem na lista. Para começar, chame /brainstorming.

Exemplo de uso na prática

Caso real: "Quero uma página onde meus clientes marcam horário comigo." Veja como cada etapa acontece — você só conversa e aprova.

  1. Descrever a ideia — /brainstorming

    VocêQuero uma página onde meus clientes marcam horário comigo.
    AgenteIsso é um app de navegador, então a base será Next.js + TypeScript. Primeira pergunta: os clientes precisam criar uma conta para marcar, ou basta nome e telefone?
    VocêBasta nome e telefone.
    AgenteComo tem dados pessoais, vou marcar isso como sensível. Aqui está o desenho em 3 partes… (apresenta e pede aprovação)

    Saída: o desenho aprovado fica salvo em docs/specs/AAAA-MM-DD--design.md.

  2. Montar o plano — /writing-plans

    O agente transforma o desenho em tarefas pequenas e verificáveis, com tarefas de segurança explícitas (validar nome/telefone, guardar dados com cuidado). Você lê em linguagem simples e aprova. Salvo em docs/plans/AAAA-MM-DD--plan.md.

  3. Construir com testes — /test-driven-development

    AgenteEscrevi um teste que confirma que marcar um horário ocupado é recusado. Rodei: ele falha (a regra ainda não existe). Agora construo o mínimo para passar…
    AgenteTeste passando ✅ e os demais continuam verdes. Essa parte impede agendamento duplicado.

    Ciclo vermelho → verde → limpar, parte por parte, tudo tipado.

  4. Revisar — /requesting-code-review

    Uma revisão independente compara o feito com o plano e faz o passo de segurança. Exemplo de retorno: "1 importante: o telefone não estava sendo validado — corrigido. Nenhum segredo no código."

  5. Entregar — /finishing-a-development-branch

    AgenteTodos os testes passam. O que você quer fazer?
    1. Juntar ao projeto · 2. Abrir Pull Request · 3. Deixar como está · 4. Descartar
    VocêOpção 2.

    O agente nunca faz commit, push ou merge sem essa sua escolha explícita.
